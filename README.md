# LLM Judge Agent System for Banking Complaint Evaluation

## Overview

This system implements an **LLM-as-a-Judge** framework with **Agentic AI** for evaluating and scoring complaint summaries generated by the SCRIBE summarizer in the banking domain. The system uses multiple specialized AI agents to classify, evaluate, and make recommendations about complaint summaries, with human-in-the-loop integration for high-risk cases.

## Features

### ✅ Core Capabilities

- **Multi-Agent Architecture**: Specialized agents for classification, evaluation, judgment, and human review
- **Configurable Evaluation**: YAML-based configuration for business rules, metrics, and prompts
- **Dynamic Metric Selection**: Complaint-type-specific evaluation metrics with configurable weights
- **Aggregate Scoring**: Weighted scoring based on metric importance
- **Risk Assessment**: Automatic risk level determination and escalation
- **Human-in-the-Loop**: Automatic generation of human review requests for high-risk complaints
- **Business Rule Engine**: Flexible rule evaluation for automated decision-making

### 📋 Complaint Types Supported

- Fraud & Unauthorized Transactions
- Account Access Issues
- Fees & Charges Disputes
- Loan & Mortgage Issues
- Customer Service Experience
- Transaction Processing Errors
- Credit Card Issues
- Regulatory & Compliance Concerns

## Configuration

### Business Rules (`config/business_rules.yaml`)

Define complaint types, risk levels, and business logic rules:

```yaml
complaint_types:
  fraud:
    risk_level: high
    priority: 1
    escalation_threshold: 0.7
    requires_human_review: true
```

### Evaluation Profiles (`config/evaluation_profiles.yaml`)

Configure metrics and weights for each complaint type:

```yaml
evaluation_profiles:
  fraud:
    metrics:
      - metric_id: "accuracy"
        name: "Factual Accuracy"
        weight: 0.30
      - metric_id: "completeness"
        name: "Information Completeness"
        weight: 0.25
```

### Prompts (`config/prompts.yaml`)

Customize LLM prompts for each agent:

```yaml
prompts:
  complaint_classifier:
    system_prompt: |
      You are an expert banking complaint classifier...
    user_prompt_template: |
      Complaint Summary: {complaint_summary}
```

## Usage

### Basic Usage

```python
from src.main import LLMJudgeOrchestrator, create_sample_complaint

# Initialize orchestrator with GPT-4 Turbo
orchestrator = LLMJudgeOrchestrator(
    model_name="gpt-4-turbo",
    output_dir="./output"
)

# Create a complaint
complaint = create_sample_complaint(
    complaint_id="COMP_001",
    complaint_type="fraud",
    summary="Customer reports unauthorized transactions...",
    original="I never made these transactions..."
)

# Evaluate
result = orchestrator.evaluate_complaint(complaint)

# Access results
evaluation = result['evaluation_result']
print(f"Score: {evaluation.weighted_score}/5")
print(f"Risk: {evaluation.risk_level.value}")
print(f"Recommendation: {evaluation.recommendation.value}")

# Check if human review needed
if result['human_review_request']:
    review = result['human_review_request']
    print(f"Human review required - Priority: {review.priority_level}")
```

### Run Example

```powershell
python run.py
```

This will evaluate sample complaints across different types and generate output files.

### Output Format

The system provides two levels of reporting:

#### 1. **Comprehensive Complaint-Wise Results Table**

Each complaint receives a detailed breakdown showing:

- **Basic Information**: Complaint ID, type, summary, submission date, customer ID
- **Metric Scores**: Individual metric scores (1-5) with weights and contributions
- **Rating Indicators**: Visual quality indicators (⭐ Excellent, ✓ Good, ○ Satisfactory, ⚠ Poor, ✗ Inadequate)
- **Aggregate/Weighted Scores**: Both unweighted and weighted final scores
- **Risk & Decision**: Risk level with indicators (🟢 Low, 🟡 Medium, 🔴 High, ⛔ Critical)
- **Recommendation**: Final decision (✅ Approve, 📝 Revise, ⬆️ Escalate, ❌ Reject)
- **Human Review Status**: Whether manual review is required (⚠️ Yes / ✓ No)
- **Confidence Level**: LLM confidence in the evaluation (percentage)
- **Strengths & Weaknesses**: Identified positive and negative aspects
- **Improvement Suggestions**: Actionable recommendations

#### 2. **Quick Reference Summary**

A condensed table showing all evaluations at a glance:

- Complaint ID and Type
- Weighted Score (0-5)
- Risk Level
- Final Decision
- Review Requirement
- Confidence Score

Both tables use visual indicators for quick scanning and decision-making.

## Evaluation Workflow

1. **Classification**: Complaint is classified into appropriate type(s)
2. **Metric Selection**: Type-specific evaluation metrics are loaded
3. **Metric Evaluation**: Each metric is scored individually by LLM
4. **Aggregate Scoring**: Weighted average calculated based on metric weights
5. **Risk Assessment**: Risk level determined based on type and scores
6. **Business Rules**: Rules evaluated to determine actions
7. **Recommendation**: Final recommendation (approve/revise/escalate/reject)
8. **Human Review**: If required, detailed review request generated

## Scoring System

- **Individual Metrics**: Scored 1-5
- **Aggregate Score**: Unweighted average of all metrics
- **Weighted Score**: Weighted average based on metric importance (0-5)
- **Risk Score**: 0-1 scale based on complaint type and quality

### Score Interpretation

- **5**: Excellent - Exceeds expectations
- **4**: Good - Meets expectations
- **3**: Satisfactory - Minimum requirements met
- **2**: Poor - Below expectations
- **1**: Inadequate - Fails basic requirements

## Human-in-the-Loop

High-risk complaints automatically trigger human review based on:

- Complaint type configuration (`requires_human_review: true`)
- Risk level (Critical or High)
- Business rule triggers (BR001, BR002, BR003)
- Low confidence scores (< 0.6)

Review requests include:

- Executive summary
- Key concerns requiring judgment
- Specific review questions
- Recommended actions
- Priority level and deadline

## Customization

### Adding New Complaint Types

1. Add to `config/business_rules.yaml`:

```yaml
complaint_types:
  new_type:
    name: "New Complaint Type"
    risk_level: medium
    priority: 3
    escalation_threshold: 0.75
```

2. Add metrics to `config/evaluation_profiles.yaml`:

```yaml
evaluation_profiles:
  new_type:
    metrics:
      - metric_id: "accuracy"
        weight: 0.30
```

3. No code changes required!

### Adding New Business Rules

Add to `config/business_rules.yaml`:

```yaml
business_rules:
  - rule_id: "BR006"
    name: "Custom Rule"
    condition: "complaint_type == 'new_type'"
    action: "custom_action"
```

### Modifying Prompts

Edit `config/prompts.yaml` to customize LLM behavior without changing code.

## Troubleshooting

### Configuration Errors

- Check YAML syntax is valid
- Ensure all required fields are present
- Verify metric weights sum to ~1.0 for each complaint type

### LLM API Errors

- **API Key Issues**: Verify `.env` file exists and contains valid `OPENAI_API_KEY`
- **Network connectivity**: Check internet connection and firewall settings
- **Rate limits**: Monitor OpenAI API usage limits and quotas
- **Model availability**: Ensure you have access to GPT-4 Turbo in your OpenAI account
- **Billing**: Verify your OpenAI account has active billing enabled

### Low Quality Scores

- Review and refine prompts in `prompts.yaml`
- Adjust metric weights in `evaluation_profiles.yaml`
- Consider different LLM models or temperature settings
